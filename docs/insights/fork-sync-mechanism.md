# Fork 同步机制的产品思考

## 用户问题

这个 fork 面临的真实问题不是“某一次 upstream 合并麻烦”，而是：

- upstream 更新很快
- fork 有自己明确的增量能力
- 这些增量能力又压在少数高频变动的核心文件上

结果就是每次同步都像重新打一场仗：

- 先看一大堆 diff
- 再靠记忆判断哪些是 fork 核心能力
- 再手工 patch
- 再担心下一次全都重来

这套方式在版本更新慢的时候还能凑合，但在 `0.50.x` 这种节奏下必然崩。

## 为什么不能继续“手工同步”

手工同步的问题不只是慢，而是不可预测：

- 同一个人隔两周再看也会忘记当初为什么这么改
- AI 或新接手开发者没有“共享的边界感”
- 每次都会重新陷入“这文件到底归官方还是归 fork”的争论

本质上，这是把“系统规则”交给了“个人经验”。

## 为什么要先做机制，而不是继续追功能

如果不先把同步机制搭起来：

- 今天补 `0.50.3`
- 明天又追 `0.50.4`
- 后天再追 `0.51`

每次都还是同样的大工作量。

所以同步机制不是附属品，而是 fork 项目能否长期活下去的前置条件。

## 本机制的核心价值

### 1. 让“差异”从隐性变显性

以前：

- 差异存在代码里
- 只有熟悉仓库的人心里知道

现在：

- ownership map 明确告诉你路径归属
- patch manifest 明确告诉你哪些能力是 fork 核心

这会显著降低“靠记忆维护 fork”的风险。

### 2. 让同步从“临场发挥”变成“有预案执行”

以前同步：

- 先开 diff
- 再猜
- 再试

现在同步：

- 先跑 bootstrap
- 看报告
- 先处理 upstream-only
- 再处理 both-changed

这能把同步成本稳定在一个可预期范围内。

### 3. 保护真正有价值的 fork 能力

你这个 fork 的价值不在于“把官方改得面目全非”，而在于一些明确的增量能力：

- 文件树
- 顶部标签和工作区
- 终端/控制台
- Git 面板
- 媒体中转
- CC Switch / OLMX

如果没有 patch manifest，这些能力迟早会再次散落到官方主链路里，下一次同步又变成全仓冲突。

### 4. 为以后接入 CI 做准备

现在先落的是：

- 边界检查
- 报告脚本
- bootstrap 工作流

下一步就能很自然地接到：

- pre-commit
- CI
- PR 检查

让“不要把 fork 逻辑继续压回主链路”变成自动化约束，而不是口头约束。

## 为什么不是直接做自动 merge

自动 merge 看起来更省事，但当前不适合直接上：

- 共享接缝文件仍然存在真实语义冲突
- 很多差异不是简单文本冲突，而是产品决策差异

所以现在最合理的路线不是“自动合并一切”，而是：

- 先把自动分类、自动预警、自动边界检查做起来
- 让人工只处理真正需要判断的少量地方

这是更现实、更稳妥的工程路径。

## 已知局限

- 机制能显著降低同步成本，但还不能把同步变成零成本
- `chat/runtime/provider` 这条主链路仍然是长期热点
- patch manifest 目前还需要人工维护

但它已经把问题从“完全不可控”降到了“可治理”。

## 下一阶段应该追求什么

机制落地后，下一阶段目标不再是继续堆功能，而是：

1. 继续把 fork 能力从主链路往外抽
2. 把 ownership 检查接进 CI
3. 让 shared seam 真正变薄
4. 让每次 upstream 更新都先跑脚本、再看少量热点文件

## 一句话总结

这套机制的价值不是“这次同步更快”，而是：

**让这个 fork 从“每次升级都要重新打一仗”，变成“有规则、有工具、有边界地长期追官方版本”。**
